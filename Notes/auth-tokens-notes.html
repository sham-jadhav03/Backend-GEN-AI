01
Why Tokens Exist
The problem with sessions — and why JWTs alone aren't enough either

Problem 1 — Traditional Sessions Don't Scale

Old-school session-based auth stores user state on the server. Every logged-in user has a session record in memory or a database. Works fine for one server. Falls apart the moment you scale to multiple servers — Server B has no idea about the session created on Server A.

Session-based auth with 3 servers:
 User logs in → hits Server A → session stored in Server A's memory
 Next request → load balancer sends to Server B
 Server B has no session → user is logged out ✗

Fix attempt: shared Redis session store
 Now every request hits Redis → bottleneck → extra infra → extra cost ✗

JWT-based auth:
 Token is self-contained → any server can verify it → no shared state needed ✅

Problem 2 — One Long-Lived JWT is Dangerous

The naive solution is to give users one JWT that lasts 30 days. Simple — but catastrophic if stolen. Once an attacker has that token, they have 30 days of full access. You can't invalidate a JWT server-side without bringing back a denylist — which defeats the whole point of stateless auth.

Single long-lived JWT (naive approach):
 Token stolen via XSS or network sniff
 Attacker has full access for 30 days
 You can't revoke it without a server-side denylist
 Stateless auth is now stateful again ✗

Two-token approach (access + refresh):
 Access token expires in 15 minutes → stolen token useless quickly ✅
 Refresh token stored in httpOnly cookie → not accessible to JS at all ✅
 Refresh tokens stored in DB → can be revoked instantly ✅
The Core Insight

The two-token pattern is a damage limitation strategy. We accept that access tokens might get stolen. So we make them expire in 15 minutes — the damage window is tiny. The refresh token, which has the real power to create new access tokens, never touches JavaScript — it lives only in an httpOnly cookie that no script can read.

02
Attacks This Pattern Prevents
Six real attacks — and exactly how the two-token pattern stops each one
💉
XSS Token Theft
An attacker injects malicious JavaScript into your page. The script runs localStorage.getItem('token') and sends your access token to their server. Done — they have full access.
Without protection: attacker gets your JWT ✗
🛡️
XSS — How We Block It
Refresh token stored in httpOnly cookie — JavaScript literally cannot read it. Access token kept in memory (React state), not localStorage. Even if XSS runs, no persistent token is stolen.
httpOnly cookie → JS can't read it ✅
🔗
CSRF Attack
Attacker tricks your browser into making a request to your API from a malicious site. The browser automatically sends your cookies — including the refresh token — so the attacker can mint new tokens.
Without protection: CSRF can trigger /refresh ✗
🛡️
CSRF — How We Block It
Set SameSite=Strict on the refresh token cookie. The browser will NOT send this cookie on cross-site requests. A CSRF attack from evil.com cannot trigger your /refresh endpoint.
SameSite=Strict → cross-site blocked ✅
⏱️
Stolen Access Token
An attacker intercepts a network request (man-in-the-middle, leaked log, misconfigured CDN) and gets the raw access token. On a 30-day token, they have weeks of access.
30-day token stolen = 30 days of access ✗
🛡️
Stolen Token — How We Limit Damage
Access token expires in 15 minutes. Even if stolen, it's useless within 15 minutes. The attacker can't get a new one because the refresh token is in an httpOnly cookie they can't read.
15-min window → damage is minimal ✅
🔄
Refresh Token Reuse Attack
Attacker steals the refresh token. They use it to get a new access token. But the real user is still active and also uses the refresh token — now both have valid sessions simultaneously.
Two users with same session — attacker persists ✗
🛡️
Reuse — Token Rotation
Each /refresh call issues a brand new refresh token AND invalidates the old one. If an old token is reused, the server detects the anomaly and invalidates the entire family — logging out everyone including the attacker.
Rotation + family invalidation → both kicked out ✅
03
Access Token vs Refresh Token
Two tokens, two jobs, two completely different security properties
🔑 Access Token
PurposeProve identity on API calls
Expiry15 minutes
Stored inMemory (React state)
Sent asAuthorization header
Stored server-side?No — stateless
Revocable?No — wait for expiry
🔄 Refresh Token
PurposeGet a new access token
Expiry7–30 days
Stored inhttpOnly cookie
Sent asCookie (automatic)
Stored server-side?Yes — in database
Revocable?Yes — delete from DB ✅
💡

Why short access token + long refresh token? The access token is used constantly — every API call. It's in memory (slightly vulnerable). Keeping it short limits the damage window. The refresh token is used rarely — only when the access token expires. It's in an httpOnly cookie (much safer). So it can live longer without significant risk.

What's inside a JWT access token

JWT decoded — three parts
// Header (algorithm + type)
{
  "alg": "HS256",
  "typ": "JWT"
}

// Payload (claims — this is what you put in)
{
  "sub":   "64f2a...",   // user's MongoDB _id
  "email": "user@example.com",
  "role":  "user",
  "iat":   1716400000,  // issued at (Unix timestamp)
  "exp":   1716400900   // expires at (15 min later)
}

// Signature — server verifies this with the secret key
// HMACSHA256(base64(header) + "." + base64(payload), SECRET)

// ⚠️ Payload is base64 encoded — NOT encrypted!
// Anyone can decode it. Never put passwords or sensitive data in JWT.

models/User.js
import mongoose from 'mongoose';
import bcrypt   from 'bcryptjs';

const userSchema = new mongoose.Schema({
  name:     { type: String, required: true },
  email:    { type: String, required: true, unique: true, lowercase: true },
  password: { type: String, required: true, select: false },
}, { timestamps: true });

// Hash password before saving
userSchema.pre('save', async function() {
  if (!this.isModified('password')) return;
  this.password = await bcrypt.hash(this.password, 12);
});

// Method to compare passwords
userSchema.methods.comparePassword = async function(candidate) {
  return bcrypt.compare(candidate, this.password);
};

export default mongoose.model('User', userSchema);

Step 5 — Token utility functions
utils/tokens.js
import jwt    from 'jsonwebtoken';
import crypto from 'crypto';

export const generateAccessToken = (user) =>
  jwt.sign(
    { sub: user._id, email: user.email, role: user.role || 'user' },
    process.env.ACCESS_TOKEN_SECRET,
    { expiresIn: process.env.ACCESS_TOKEN_EXPIRY }
  );

export const generateRefreshToken = () =>
  crypto.randomBytes(64).toString('hex');
// Refresh token is a random string — NOT a JWT
// We look it up in the DB, no payload needed

export const generateTokenFamily = () =>
  crypto.randomBytes(16).toString('hex');
// Family ID — all refresh tokens from one login share this
// If an old token from this family is reused → revoke entire family

export const setRefreshCookie = (res, token) => {
  res.cookie('refreshToken', token, {
    httpOnly:  true,  // JS can't read it
    secure:    process.env.NODE_ENV === 'production', // HTTPS only in prod
    sameSite:  'strict', // blocks CSRF
    maxAge:    7 * 24 * 60 * 60 * 1000, // 7 days in ms
    path:      '/api/auth', // cookie only sent to auth routes
  });
};

Auth Routes
Register, login, refresh, and logout — complete implementations
Register
routes/auth.js — POST /auth/register
import express      from 'express';
import User          from '../models/User.js';
import RefreshToken  from '../models/RefreshToken.js';
import { generateAccessToken, generateRefreshToken,
         generateTokenFamily, setRefreshCookie } from '../utils/tokens.js';

const router = express.Router();

router.post('/register', async (req, res) => {
  try {
    const { name, email, password } = req.body;
    const user = await User.create({ name, email, password });

    const accessToken  = generateAccessToken(user);
    const refreshToken = generateRefreshToken();
    const family       = generateTokenFamily();

    await RefreshToken.create({
      token:     refreshToken,
      userId:    user._id,
      family,
      expiresAt: new Date(Date.now() + 7 * 24 * 60 * 60 * 1000),
    });

    setRefreshCookie(res, refreshToken);
    res.status(201).json({ accessToken, user: { id: user._id, name, email } });
  } catch (err) {
    res.status(400).json({ error: err.message });
  }
});
Login
POST /auth/login
router.post('/login', async (req, res) => {
  try {
    const { email, password } = req.body;

    // select('+password') because we used select: false in schema
    const user = await User.findOne({ email }).select('+password');
    if (!user || !(await user.comparePassword(password))) {
      return res.status(401).json({ error: 'Invalid credentials' });
    }

    const accessToken  = generateAccessToken(user);
    const refreshToken = generateRefreshToken();
    const family       = generateTokenFamily();

    await RefreshToken.create({
      token: refreshToken, userId: user._id,
      family, expiresAt: new Date(Date.now() + 7*24*60*60*1000),
    });

    setRefreshCookie(res, refreshToken);
    res.json({ accessToken, user: { id: user._id, name: user.name, email } });
  } catch (err) {
    res.status(500).json({ error: err.message });
  }
});
Refresh — with token rotation
POST /auth/refresh
router.post('/refresh', async (req, res) => {
  try {
    const incomingToken = req.cookies?.refreshToken;
    if (!incomingToken) return res.status(401).json({ error: 'No refresh token' });

    // Find the token in DB
    const stored = await RefreshToken.findOne({ token: incomingToken });

    if (!stored) {
      // Token not found — could be a reuse attack
      // Try to find if this family still exists and nuke it
      res.clearCookie('refreshToken');
      return res.status(403).json({ error: 'Token reuse detected' });
    }

    // Check expiry
    if (stored.expiresAt < new Date()) {
      await RefreshToken.deleteOne({ _id: stored._id });
      res.clearCookie('refreshToken');
      return res.status(403).json({ error: 'Refresh token expired' });
    }

    // ── TOKEN ROTATION ───────────────────────────────
    // Delete the old token
    await RefreshToken.deleteOne({ _id: stored._id });

    // Issue a new pair
    const user         = await User.findById(stored.userId);
    const accessToken  = generateAccessToken(user);
    const newRefresh   = generateRefreshToken();

    await RefreshToken.create({
      token:     newRefresh,
      userId:    user._id,
      family:    stored.family, // keep same family for attack detection
      expiresAt: new Date(Date.now() + 7*24*60*60*1000),
    });

    setRefreshCookie(res, newRefresh);
    res.json({ accessToken });
  } catch (err) {
    res.status(500).json({ error: err.message });
  }
});
Logout
POST /auth/logout
router.post('/logout', async (req, res) => {
  const token = req.cookies?.refreshToken;
  if (token) {
    await RefreshToken.deleteOne({ token }); // invalidate server-side
  }
  res.clearCookie('refreshToken', { path: '/api/auth' });
  res.json({ message: 'Logged out' });
});

export default router;

Auth Middleware
Protect routes by verifying the access token on every request
middleware/auth.js
import jwt from 'jsonwebtoken';

export const protect = (req, res, next) => {
  const authHeader = req.headers.authorization;

  if (!authHeader?.startsWith('Bearer ')) {
    return res.status(401).json({ error: 'No access token provided' });
  }

  const token = authHeader.split(' ')[1];

  try {
    const decoded = jwt.verify(token, process.env.ACCESS_TOKEN_SECRET);
    req.user = decoded;  // { sub, email, role, iat, exp }
    next();
  } catch (err) {
    if (err.name === 'TokenExpiredError') {
      return res.status(401).json({ error: 'Access token expired', code: 'TOKEN_EXPIRED' });
    }
    res.status(403).json({ error: 'Invalid token' });
  }
};

// Role-based access control — use after protect
export const restrictTo = (...roles) => (req, res, next) => {
  if (!roles.includes(req.user.role)) {
    return res.status(403).json({ error: 'Insufficient permissions' });
  }
  next();
};
Protect a route
routes/user.js

React Frontend
Auth context, axios interceptor for silent refresh, and protected routes
Auth Context — store token in memory
context/AuthContext.jsx
import { createContext, useContext, useState, useEffect } from 'react';
import api from '../api/axios.js';

const AuthContext = createContext(null);

export const AuthProvider = ({ children }) => {
  // ✅ Token lives in state — NOT localStorage
  const [accessToken, setAccessToken] = useState(null);
  const [user, setUser]               = useState(null);
  const [loading, setLoading]         = useState(true);

  // On app mount — try to restore session silently
  useEffect(() => {
    const restoreSession = async () => {
      try {
        // httpOnly cookie sent automatically
        const { data } = await api.post('/auth/refresh');
        setAccessToken(data.accessToken);
        setUser(data.user);
      } catch {
        // No valid refresh token — user must log in
      } finally {
        setLoading(false);
      }
    };
    restoreSession();
  }, []);

  const login = async (email, password) => {
    const { data } = await api.post('/auth/login', { email, password });
    setAccessToken(data.accessToken);
    setUser(data.user);
    return data;
  };

  const logout = async () => {
    await api.post('/auth/logout');
    setAccessToken(null);
    setUser(null);
  };

  return (
    <AuthContext.Provider value={{ accessToken, setAccessToken, user, login, logout, loading }}>
      {!loading && children}
    </AuthContext.Provider>
  );
};

export const useAuth = () => useContext(AuthContext);
Axios instance with silent refresh interceptor
api/axios.js
import axios from 'axios';

const api = axios.create({
  baseURL:         'http://localhost:5000/api',
  withCredentials: true, // send cookies on every request
});

// REQUEST interceptor — attach access token
api.interceptors.request.use((config) => {
  // Get token from auth context (we'll set this up below)
  const token = window.__accessToken__;
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});

// RESPONSE interceptor — silent refresh on 401
api.interceptors.response.use(
  (response) => response, // pass through success
  async (error) => {
    const original = error.config;

    // If 401 and we haven't retried yet
    if (error.response?.status === 401 && !original._retry) {
      original._retry = true;
      try {
        // Get a new access token silently
        const { data } = await api.post('/auth/refresh');

        // Update the global token reference
        window.__accessToken__ = data.accessToken;

        // Retry the original request with new token
        original.headers.Authorization = `Bearer ${data.accessToken}`;
        return api(original);
      } catch {
        // Refresh failed — session truly expired, redirect to login
        window.location.href = '/login';
      }
    }
    return Promise.reject(error);
  }
);

export default api;

Where to Store Tokens
The most debated question in frontend auth — and the correct answer
StorageXSS riskCSRF riskVerdict
localStorage (access token) 🔴 High — any JS can read it 🟢 None — not a cookie Never. XSS = full token theft.
Memory / React state (access token) 🟢 Low — wiped on tab close 🟢 None ✅ Correct for access tokens
httpOnly cookie (refresh token) 🟢 None — JS cannot read it 🟡 Possible — use SameSite=Strict ✅ Correct for refresh tokens
The Rule

Access token → memory only. Short-lived, lives in React state, gone when tab closes. Refresh token → httpOnly cookie only. JavaScript literally cannot read httpOnly cookies — it's a browser-level restriction, not a JavaScript one. This is the only combination that protects against both XSS and CSRF simultaneously.

⚠️

Never store JWTs in localStorage. This is the single most common security mistake in MERN apps. Every security audit flags it. If an attacker injects one line of JavaScript via XSS, a malicious npm package, or a third-party script, they run localStorage.getItem('token') and immediately exfiltrate every user's token. The fix is one line: use memory instead.

The complete pattern — summary

WhatHowWhy
Access tokenMemory (React useState). Sent as Authorization: Bearer header.JS-readable but short-lived. Even if stolen, expires in 15 min.
Refresh tokenhttpOnly cookie. SameSite=Strict, Secure in prod.JS can't read it. CSRF blocked by SameSite. Revocable in DB.
Token rotationNew refresh token on every /refresh. Old one deleted immediately.Stolen refresh token detected if used after rotation.
